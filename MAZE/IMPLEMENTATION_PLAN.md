# MAZE Web Game Implementation Plan

## Project Overview

Transform Christopher Manson's MAZE puzzle book into an interactive text-based web game. Players will navigate through 45 interconnected rooms, encountering cryptic text and navigating based on door choices. The game recreates the puzzle-solving experience of the original book in an accessible web format.

**Target**: Faithful adaptation capturing the mysterious, non-linear exploration and puzzle-solving nature of MAZE.

## Core Game Systems

### 1. Data Structure

The game uses raw data from `results.json` generated by the Playwright scraper:
```javascript
{
  "Room-01": {
    "paragraphs": ["text line 1", "text line 2", ...],
    "nextRooms": ["Room-21", "Room-41", "Room-20", "Room-26"]
  },
  "Room-02": { ... },
  // ... 47 rooms total (Directions, Prologue, Room-01 through Room-45)
}
```

**Key Properties**:
- 47 rooms total (Directions intro + Prologue + 45 numbered rooms)
- Each room has text paragraphs and list of accessible next rooms
- Room navigation defined by `nextRooms` array
- Room 45 is the final destination (center of maze)
- Room 24 is a dead-end trap (empty nextRooms array)

### 2. Game State Management

**Game State**:
```javascript
{
  currentRoom: "Room-01",              // String key from results.json
  visitedRooms: Set<string>,          // Tracks all visited rooms
  moveCount: number,                   // Total moves made
  hintsUsed: number,                   // Count of hints revealed
}
```

**Navigation Logic**:
- Player enters room name/number
- Check if room exists in `nextRooms` array of current room
- If valid, transition to new room:
  1. Print the last paragraph of current room (narrative bridge)
  2. Update image to destination room
  3. Print all paragraphs from destination room with typewriter effect
  4. Update currentRoom, add to visitedRooms, increment moveCount
- If invalid: `> There is no door to Room-XX from here.` (includes attempted room number, but do not reveal which rooms are valid)
- Detect revisits: if currentRoom already in visitedRooms, prompt user before reprinting text
- Dead-end detection: if `nextRooms` is empty, show special message (e.g., Room 24)

**Hint System** (Level 1 Only - MVP):
- **Hint**: List all rooms player hasn't visited yet

**Future Hint Levels** (TBD):
When implementing deeper hint levels, ensure they only reveal information the player has already discovered through gameplay. Options to explore:
- Showing available doors from current room
- Identifying dead-end rooms
- Shortest path guidance (if player has explored enough)
- Avoid revealing room content or story elements player hasn't encountered

**CRITICAL**: Do NOT provide hints about Room 29's hidden door puzzle. Room 29 contains a secret/hidden door that leads to Room 17 (visible only through the image). Discovering this hidden passage is a key architectural element of the maze difficulty and must be discovered by the player.

### 3. Terminal UI System (Retro Command-Line Aesthetic)

**Design Philosophy**:
Recreate the feel of 1980s text-based adventure games with authentic terminal aesthetics while maintaining full mobile responsiveness.

**Visual Theme**:
- **Text Color**: Bright green (#00FF00 or #00DD00) on pure black (#000000)
- **Font**: Monospace (Courier New, Courier Prime, or system monospace)
- **Font Size**: Responsive scaling (18-24px on desktop, 14-16px on mobile)
- **Text Rendering**: Anti-aliased for readability at smaller sizes
- **Cursor**: Blinking underscore or block character animation

**Main Interface Layout** (Responsive Grid):
```
┌──────────────────────────────────────────────┐
│ [IMAGE: Room displayed at top, full width]  │
│ (Scales responsively, maintains aspect ratio)│
├──────────────────────────────────────────────┤
│ > {{ last line from previous page }}         │
│ > ...the entrance hall of the Maze.         │
│ > They looked carefully at the bronze doors │
│ > trying to choose...                       │
│                                             │
│ > : _                                        │
└──────────────────────────────────────────────┘
```

**Components**:
1. **Image Viewport**: Room image at top, responsive width, contained within terminal
2. **Text Viewport**: Character-by-character typewriter output of room description
3. **Command Line**: Monospace input area with blinking cursor
4. **Navigation Buttons**: Door commands as green text buttons ([GO 21], [GO 41], etc.)
5. **System Commands**: Help, hints, inventory, status as text buttons
6. **Status Line**: Current room, move count, hint count in terminal format

**Typewriter Effect Implementation**:
- Each character appears with 30-50ms delay
- Sound effect (optional): Subtle typing sound on each character
- User can press SPACE/ENTER to skip typewriter animation
- Smooth scrolling as text fills viewport
- Text wraps at terminal width

**Mobile Responsiveness**:
- **Viewport Units**: Uses 100vh/100vw with no scrolling (fills entire screen)
- **Breakpoints**:
  - Desktop (1024px+): Full layout with proportional image
  - Tablet (768px-1024px): Balanced layout
  - Mobile (<768px): Compressed layout, optimized spacing
- **Touch Targets**: All command buttons minimum 44px height for mobile
- **Image Scaling**: Uses CSS `object-fit: contain` to maintain aspect ratio
- **Font Scaling**: Responsive sizing with clamp() function

**Command System**:
- **Navigation**: `GO [room-number]` (e.g., `GO 21`)
- **Help**: `HELP` - display available commands and syntax
- **Hints**: `HINT` - show unexplored rooms (non-spoiling hint)
- **Status**: `STATUS` - show current room, move count, visited rooms
- **Clear**: `CLEAR` - clear screen (optional)

**Animation & Feedback**:
- Text appears character-by-character (typewriter effect)
- Door transitions: 
  1. Print last paragraph of current room with typewriter effect (narrative bridge to next page)
  2. Update image viewport to destination room
  3. Print all paragraphs from destination room with typewriter effect
  4. This flow maintains narrative continuity between rooms
- Hint reveal: Progressive disclosure with typewriter effect
- Error messages: Narrative responses that include the attempted command or room:
  - Invalid room: `> There is no door to room XX from here.`
  - Invalid command: `> 'COMMAND' is not something you can do here.`
- Victory: Special screen with ASCII celebration graphics

## Technical Implementation Details

### Game State Management
```javascript
// Using React Context for global state
const GameState = {
  rooms: {},              // All room data
  currentRoom: 1,
  visited: new Set(),
  inventory: [],
  moves: 0,
  hints: 0,
  maxHints: 3,
  path: [],               // Breadcrumb trail
};

// Actions
dispatch({ type: 'MOVE_TO_ROOM', payload: roomId })
dispatch({ type: 'USE_HINT', payload: roomId })
dispatch({ type: 'COLLECT_ITEM', payload: item })
dispatch({ type: 'RESET_GAME' })
```

### Room Navigation Validation
```javascript
function validateDoorChoice(currentRoom, chosenDoor) {
  const room = rooms[currentRoom];
  // Check if chosenDoor exists in the room's nextRooms array
  if (room.nextRooms.includes(chosenDoor)) {
    return { valid: true, nextRoom: chosenDoor };
  }
  return { valid: false, message: "There is no door to that room from here." };
}
```

### Hint System (Non-Spoiling)
```javascript
// Hint provides information player has already discovered
// Level 1 hints only: show unvisited rooms (based on game state)
function getHint(gameState) {
  const allRooms = new Set(Object.keys(rooms));
  const unvisitedRooms = [...allRooms].filter(room => !gameState.visitedRooms.has(room));
  return "Rooms you haven't explored yet: " + unvisitedRooms.join(", ");
}
```
